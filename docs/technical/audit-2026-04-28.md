## Contexte

Audit déclenché avant la phase finale du jalon 15 (yOffset livré) et le démarrage du jalon 16 (mode unifié, refonte persistence, undo/redo, routing, redesign drawer/CSV).

Objectif : décider entre **refacto incrémental** sur la branche `dev` et **réécriture from scratch** dans un nouveau repo, avant d'engager les ~8 sessions de travail prévues.

## Méthode

Trois sous-agents `Explore` lancés en parallèle, périmètre `src/` :

2. Couplage (leaks de couche, hotspots Redux, code mort).
3. Tests et dépendances (couverture, JSDOM, libs runtime).

### Couplage et leaks

- **Architecture 85 % propre**. Frontières `core` / `persistence` / `store` / `components` globalement respectées.
- **Leaks identifiés** :
  - `persistence/db.ts` importe les types `*Record` depuis `@/store/types` → relation circulaire à inverser (les `*Record` appartiennent à `persistence/`).

  - 5 composants (`ThemeSwitcher`, `NewProjectDialog`, `HamburgerMenu`, `CloneProjectDialog`, `DeleteProjectConfirm`) importent directement `@/persistence/lastProjectId` → devrait passer par store/thunk.
  - `main.tsx` applique le thème via `applyTheme()` (persistence) au boot.
- **Hotspot orchestrateur** : `components/canvas/Row/index.tsx` consomme 10 hooks Redux (8 selectors + 2 dispatch). Candidat principal au découpage props in / callbacks out.
- **Code mort** : `core/geometry.ts:roundToTenth()` exporté, zéro consommateur.

## Recommandation : refacto incrémental sur branche `rewrite/v2`

### Pourquoi pas greenfield

Greenfield était attractif uniquement si la dette était profonde. Les chiffres montrent l'inverse :

- `core/` (1 358 LOC, 82 scénarios BDD, zéro leak) = **stable et copy-paste-ready**, mais dans **les deux scénarios**.
- `persistence/` ne fait que 252 LOC. Réécrire sur place coûte le même prix qu'à blanc.
- `store/` est de petite taille (760 LOC), reducers déjà purs, prêt pour `redux-undo`.
- **0 test JSDOM** à abandonner. Vitest config = ~10 lignes. Aucune dette test.
- Stack runtime minimale (6 libs). Pas de cleanup à faire avant d'ajouter routing/undo.
- CI/CD GitHub Actions, configs Vite/ESLint/tsconfig : à recopier identiques → coût équivalent.

Le surcoût réel d'un greenfield (re-bootstrap repo, re-config, re-CI/CD, refaire l'historique git, perdre les PR closes) n'est pas compensé par un gain de dette évitée.

### Stratégie retenue

**Branche `rewrite/v2` à partir de `main`** (pas un nouveau repo). Permet :

- Garder l'historique, les conventions, les configs.
- Itérer en sessions atomiques (1 session = 1 PR sur `rewrite/v2`).
- Merger `rewrite/v2` → `main` au bout des 8 sessions une fois la migration IDB v3→v4 livrée.
- Garder `dev` comme fallback (rollback simple).

## Stratégie tokens

Le diagnostic utilisateur (« début de session = 20 % du budget ») est traité avec :

1. **Plans pré-rédigés exhaustifs** dans `.agent/specs/16.X-*.md` avant chaque session. Une session = « exécute `16.X.md` ». Démarrage tombe à ~5 %.
2. **Sous-agents systématiques** : `Explore` pour le mapping en début de session, `Plan` pour rédiger les specs en avance, `general-purpose` pour les tâches autonomes (patcher N fixtures, par exemple). Préserve le contexte main.
3. **Séparation docs humains / agents** :
   - `docs/` reste pour les humains (features, technical, glossary, project-steps, wireframes).
   - `.agent/` (nouveau) contient `conventions.md`, `patterns/*.md`, `specs/16.X.md`, `fixtures/*`. Lecture ciblée par les agents, pas de bruit narratif.
4. **CLAUDE.md durci** : libs autorisées listées, taille fichier 150 LOC en hard limit, conventions tests « store-first / pas de JSDOM », composants découplés du store par défaut.
5. **Une session = un commit + un push + une PR** sur `rewrite/v2`. Pas de drift, pas de PR fleuves.

## Décisions à valider

1. **Refacto incrémental sur `rewrite/v2`** plutôt que greenfield → confirmation utilisateur.
2. **Périmètre des 8 sessions** ci-dessus → ajustements éventuels avant rédaction des specs.
3. **Libs externes** : `wouter` (routing, ~2 KB) + `redux-undo` (undo, ~4 KB). Pan/zoom reste maison. Confirmation utilisateur.
4. **Session de bootstrap (S0)** : rédiger les 8 specs `.agent/specs/16.X.md` + créer `.agent/conventions.md` + `.agent/patterns/`. Investissement unique, débloque les 8 suivantes.
