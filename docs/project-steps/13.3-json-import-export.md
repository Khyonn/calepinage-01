# Sous-étape 13.3 — Export / Import JSON

## 1. Objectif utilisateur

Depuis le menu hamburger, l'utilisateur peut :
- **Exporter** le projet courant dans un fichier `.json` (téléchargement navigateur) contenant l'intégralité des données nécessaires à sa reconstruction (catalogue, paramètres de pose, pièces, rangées, plan de fond image incluse en base64).
- **Importer** un fichier `.json` précédemment exporté. L'import crée **toujours un nouveau projet** (avec id régénéré), n'écrase jamais le projet courant. En cas de collision de nom, l'import appendra ` (importé)` au nom.

Ce livrable ouvre la porte au partage de projet, à la sauvegarde hors IndexedDB, et prépare le clonage depuis un projet existant (jalon 15.5).

## 4. Format du fichier JSON

```json
{
  "version": 1,
  "exportedAt": "2026-04-24T10:00:00.000Z",
  "project": {
    "id": "…",
    "name": "Appartement",
    "poseParams": { "cale": 0.5, "sawWidth": 0.1, "minPlankLength": 30, "minRowGap": 15 },
    "catalog": [
      { "id": "…", "projectId": "…", "name": "Chêne", "length": 120, "width": 14,
        "pricing": { "type": "lot", "pricePerLot": 45, "lotSize": 8 }, "description": "" }
    ],
    "rooms": [
      { "id": "…", "projectId": "…", "name": "Salon",
        "vertices": [{ "x": 0, "y": 0 }, …],
        "rows": [{ "id": "…", "roomId": "…", "plankTypeId": "…", "segments": [{ "xOffset": 0 }] }]
      }
    ],
    "backgroundPlan": {
      "id": "…", "projectId": "…",
      "calibration": { "point1": {…}, "point2": {…}, "realDistance": 200 },
      "opacity": 0.7, "rotation": 0, "x": 0, "y": 0
    }
  },
  "image": {
    "name": "plan.png",
    "mimeType": "image/png",
    "dataUrl": "data:image/png;base64,iVBORw0KGgo…"
  }
}
```

Notes :
- `project.backgroundPlan.imageFile` **absent** du JSON — l'image est dans `image` au niveau racine.
- `image` omis si le projet n'a pas de plan de fond.
- JSON indenté 2 spaces pour lisibilité humaine.
- Encodage UTF-8.

## 5. Comportement attendu

### Export

- [ ] Entrée « Exporter JSON » dans le menu hamburger, disabled si pas de projet courant.
- [ ] Clic → téléchargement immédiat `calepinage-<slug>-<date>.json` (mêmes règles de slug que l'export CSV).
- [ ] Si le projet a un `backgroundPlan.imageFile`, l'image est lue via `FileReader.readAsDataURL` et incluse dans le JSON.
- [ ] Si l'image ne peut pas être lue (erreur FileReader), l'export échoue avec une alerte utilisateur (pas d'export partiel silencieux).
- [ ] Le fichier résultant est un JSON valide (passe `JSON.parse`).

### Import

- [ ] Entrée « Importer JSON » dans le menu hamburger, toujours activée (même sans projet courant).
- [ ] Clic → ouvre un file picker (`<input type="file" accept="application/json,.json">`).
- [ ] Fichier sélectionné → parse, validation.
- [ ] Validation → version reconnue (`version === 1`), champs requis présents (`project.id`, `project.name`, `project.poseParams`, `project.catalog`, `project.rooms`). Les types sont vérifiés (tableaux, shapes d'objet).
- [ ] Régénération des ids : nouveau `project.id`, nouveaux `room.id`, `row.id`, `plankType.id`. Les références croisées (`row.plankTypeId`, `row.roomId`, `plankType.projectId`, etc.) sont remappées via `Map<oldId, newId>`.
- [ ] L'image (si présente) est re-hydratée en `File` via `fetch(dataUrl).then(r => r.blob()).then(b => new File([b], name, { type }))`.
- [ ] Le nouveau projet est créé via le thunk d'import — passe par le flux de persistance IDB normal (pas de court-circuit).
- [ ] Nom en collision : si un projet avec le même nom existe déjà → appendre ` (importé)` au nom (suffixe numérique si plusieurs collisions).
- [ ] Erreur de validation (version inconnue, champ manquant, shape cassée) → dialog d'erreur avec message explicite (« Format non supporté », « Fichier corrompu »).
- [ ] Le nouveau projet devient le projet courant (comportement cohérent avec `createProjectThunk`).
- [ ] L'utilisateur peut immédiatement voir et éditer le projet importé.

## 6. Tests

### Unitaires — `project-serialize.feature`

```gherkin
Scénario: Round-trip export → import — projet sans plan
  Étant donné un projet avec catalogue, pièces et rangées
  Quand je l'exporte en JSON puis le ré-importe
  Alors les données du projet réimporté sont structurellement identiques
    (aux ids près, qui sont régénérés)

Scénario: Round-trip avec plan de fond
  Étant donné un projet avec un plan de fond (image PNG)
  Quand je l'exporte puis le ré-importe
  Alors l'image est présente et a le même contenu binaire
    Et la calibration, l'opacité, la rotation, la position sont préservées

Scénario: Validation — version inconnue
  Quand je tente d'importer un JSON avec "version": 999
  Alors une erreur "Format non supporté" est levée

Scénario: Validation — champ requis manquant
  Quand je tente d'importer un JSON sans "project.poseParams"
  Alors une erreur "Fichier corrompu" est levée

Scénario: Régénération des ids et remap des références croisées
  Étant donné un projet avec rows pointant vers plankTypes par id
  Quand je l'importe
  Alors les ids sont tous neufs
    Et row.plankTypeId pointe bien vers le plankType remappé correspondant
```

### Test manuel

1. Créer un projet avec 2 pièces, catalogue 2 types, 4 rangées, plan de fond importé + calibré.
2. Exporter JSON → vérifier le téléchargement, ouvrir le fichier dans un éditeur.
3. Effacer IndexedDB (DevTools → Clear).
4. Recharger l'app → projet auto-créé vide.
5. Importer le JSON sauvegardé → le projet réapparaît à l'identique (y compris l'image de fond).
6. Exporter à nouveau → diff avec le fichier original : seuls les ids doivent différer.

## 8. Hors scope

- **Import qui fusionne avec le projet courant** (au lieu d'en créer un nouveau) : hors scope. Toujours créer un nouveau projet.
- **Clonage depuis un projet existant** : traité en 15.5, utilisera les primitives de remap d'ids introduites ici.
- **Partage en ligne (URL, cloud)** : hors scope. L'app reste 100 % locale.
- **Migration automatique entre versions de schéma** : `version: 1` unique pour l'instant. Une future v2 nécessitera une migration explicite.

## Notes pour l'agent

- La régénération d'ids + remap des références croisées est une primitive potentiellement réutilisée en 15.5 (clone). L'exposer via un helper pur `remapProjectIds(project) → Project` dans `projectSerialize.ts` ou un module dédié.
- Ne pas ajouter de dépendance zod / io-ts. Validation manuelle suffit pour un schéma petit et stable.
- Prendre garde aux images volumineuses : base64 gonfle +33 % ; un plan 5 Mo → JSON ~6,7 Mo. Pas de limite imposée mais logger la taille en console en cas d'export > 10 Mo.
- L'import doit passer par le flux de persistance normal (thunk) — ne pas écrire directement dans IndexedDB, pour préserver les invariants du middleware.
