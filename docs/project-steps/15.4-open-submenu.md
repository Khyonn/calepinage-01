# Sous-étape 15.4 — Ouvrir un projet via sous-menu

## 1. Objectif utilisateur

L'entrée « Ouvrir un projet » du hamburger expose directement la liste des autres projets en **sous-menu cascade** (flyout latéral) au lieu d'ouvrir un dialog modal. Un click sur un projet → switch immédiat. Plus rapide qu'un dialog dédié.

## 2. Dépendances

- Jalon 06 ✅ (HamburgerMenu, OpenProjectDialog).
- Jalon 14.1 ✅ (Tooltip — pas de dépendance directe, mais pattern portal/positionnement réutilisable).
- Précurseur de 15.5 (même primitive `ui/Menu`).

## 3. Fichiers à créer / modifier

**Créer :**
- `src/components/ui/Menu/index.tsx` — primitive cascade. Exporte `Menu`, `MenuItem`, `MenuSubmenu`. Submenu apparaît au hover/focus de l'item parent. Positionné via portal et `getBoundingClientRect()` (même pattern que `Tooltip`).
- `src/components/ui/Menu/Menu.module.css` — surface, ombre, hover state, séparateur.
- `src/components/ui/Menu/useMenuPosition.ts` — calcul flyout (à droite du parent par défaut, flip à gauche si overflow viewport).

**Modifier :**
- `src/components/layout/HamburgerMenu/index.tsx` — remplacer le `<button>` "Ouvrir un projet" + `<OpenProjectDialog>` par un `MenuSubmenu` listant `list.filter(p => p.id !== current?.id)`. Click sur un item → `dispatch(openProjectThunk(id, …))`, fermer le menu racine.
- Migrer la structure interne du HamburgerMenu vers la primitive `Menu` (cohérence). Sinon : laisser le menu actuel et ajouter juste un sous-menu inline pour minimiser le diff. Choix retenu : laisser la structure actuelle, n'introduire `MenuSubmenu` que pour le sous-menu.

**Supprimer :**
- `src/components/layout/OpenProjectDialog/index.tsx` + `.module.css` — plus aucun consommateur.

## 7. Risques & arbitrages

- **Complexité primitive Menu** : Risque d'over-engineering. Garder l'API minimale : `<Menu>{...items}</Menu>`, `<MenuItem onSelect>`, `<MenuSubmenu label>{...items}</MenuSubmenu>`. Pas de positionnement avancé (Floating UI).
- **Tactile** : pas de hover. Sur mobile, click sur l'item parent ouvre le sous-menu. Documenté hors scope si non testé.
- **Délai d'ouverture sous-menu** : 150 ms par défaut (cohérent avec patterns natifs). Évite les ouvertures intempestives au survol rapide.
- **Conserver dialog vs supprimer** : la spec demande un sous-menu en remplacement. Supprimer le dialog évite la divergence et le code mort. Si plus tard on veut un mode "liste avancée" (rechercher, trier), on rouvrira un sous-step.

## Notes pour l'agent

- Réutiliser `localStorageLastProjectId` via le thunk `openProjectThunk` comme aujourd'hui.
- `MenuSubmenu` doit utiliser `createPortal(document.body)` pour éviter les problèmes de clip/overflow du conteneur menu racine.

- Le flyout doit avoir z-index ≥ `--z-menu`. Si plusieurs niveaux : `var(--z-menu)` partout suffit (l'ordre DOM portal gère la pile).
