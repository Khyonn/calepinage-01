# Sous-étape 14.3 — Récapitulatif par rangée avec hover granulaire

## 1. Objectif utilisateur

Le drawer résumé remplace le tableau abstrait des liens de réutilisation par un **récapitulatif structuré par rangée** : pour chaque pièce du projet, on liste ses rangées, et pour chaque rangée la longueur de sa première et de sa dernière planche, accompagnées d'un **chip** indiquant l'origine ou la destination de cette planche (offcut consommé ← rangée source, ou offcut produit → rangée cible). En survolant un de ces chips, l'utilisateur voit la **planche concernée** s'illuminer dans le canvas — la première planche de la rangée locale **et** la dernière planche de la rangée distante (ou inversement) — sans flouter la rangée entière comme avant.

## 5. Détails de calcul / dérivation

**Première et dernière planche par rangée :**

Pour la rangée d'index `i` dans la pièce :
- Récupérer `geom = computeRowGeometry(room, i, catalog, poseParams)`. Si `null` → ignorer (rangée non rendable).
- `firstSeg = geom.segments[0]` ; `lastSeg = geom.segments[geom.segments.length - 1]`.
- `firstLength = firstSeg?.planks[0]?.length`. Skip si absent.
- `lastLength = lastSeg?.planks[lastSeg.planks.length - 1]?.length`. Skip si absent ou si `firstSeg === lastSeg && planks.length === 1` (alors `firstLength === lastLength`, on n'affiche que `Première`).

**Liens entrants/sortants :**

```ts
const incoming = links.find(l => l.targetRowId === row.id) ?? null
const outgoing = links.find(l => l.sourceRowId === row.id) ?? null
```

Affichage du chip : résoudre `sourceRowId` / `targetRowId` → `roomName` + `rowIndex+1`.

## 8. Risques & arbitrages

- **Granularité offcut sur rangée multi-segment** : `computeOffcutLinks` ne considère que `segment[0]` côté target et le dernier plank du fillRow de `segment[0]` côté source (cf. `rowFill.ts:104-105`). Pour la **target**, la première planche est sans ambiguïté `geom.segments[0].planks[0]`. Pour la **source**, le "dernier plank" produisant l'offcut est celui du **segment[0]**, pas du dernier segment. Highlight cohérent : pour `position: 'last'` source, viser `segments[0]`'s last plank (ne pas viser `segments[N-1]`). C'est un détail subtil — documenter dans le code.
- **Performances** : la dérivation parcourt toutes les rangées. À 100+ rangées, négligeable. Mémoïser via `createSelector`.
- **Densité visuelle** : un projet à 50+ rangées rend le drawer long. Acceptable pour ce jalon — scroll natif. Pagination/filtre hors scope.
- **Chip cliquable plus tard** : on prévoit le chip comme `<button>` ou `<span role="button">` pour l'accessibilité, mais sans `onClick` actif. Évite refactor futur si on ajoute "click → recentrer canvas sur la planche" plus tard.
- **Cas inter-pièce** : un link peut traverser deux pièces. Le chip doit alors mentionner le nom de la pièce distante. Sinon, on n'affiche que le numéro de rangée pour la concision.
