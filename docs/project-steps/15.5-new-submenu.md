# Sous-étape 15.5 — Nouveau projet : sous-menu scratch / import / clone

## 1. Objectif utilisateur

L'entrée « Nouveau projet » du hamburger devient un sous-menu cascade proposant trois points d'entrée : (a) **À partir de rien** (équivalent du formulaire actuel — un nom), (b) **Depuis un fichier JSON** (réutilise l'import 13.3), (c) **Depuis un projet existant** (clonage avec sélection des éléments à reprendre : catalogue, paramètres de pose, plan de fond, pièces, rangées). Permet à l'utilisateur de démarrer un nouveau projet en réutilisant rapidement la base technique (pose + catalogue) d'un projet existant.

## 5. Détails de calcul

**Filtrage du source avant `remapProjectIds`** :

```ts
const filtered: SerializedProject['project'] = {
  id: source.id,
  name: newName,
  poseParams: options.poseParams ? source.poseParams : DEFAULT_POSE_PARAMS,
  catalog: options.catalog ? source.catalog : [],
  rooms: options.rooms
    ? source.rooms.map(r => ({
        ...r,
        rows: options.rows ? r.rows : [],
      }))
    : [],
  ...(options.backgroundPlan && source.backgroundPlan
    ? { backgroundPlan: source.backgroundPlan }
    : {}),
}
const cloned = remapProjectIds(filtered)
```

Note : `remapProjectIds` re-génère les ids et conserve les références internes (rows.plankTypeId remappé). Si `options.rows` true mais `options.catalog` false, on a une incohérence — la cascade UI doit empêcher cette combinaison.

**Plan de fond** : `imageFile` (référence à un File) doit être recopié — `remapProjectIds` actuel ne touche pas au `imageFile`. À vérifier au moment de la persistence : la couche IDB stocke l'image dans le store `files` séparé ; un clonage du `backgroundPlan` doit dupliquer ou re-référencer l'image. Décision : pour 15.5, on duplique le File via `new File([blob], …)` si nécessaire. Tests à ajouter.

## 8. Risques & arbitrages

- **Image du plan de fond** : `imageFile` est un objet `File`. Le dupliquer suffit-il en mémoire (Object URL identique potentiellement partagé) ? Décision : `cloneProject` reçoit le `Project` runtime — le champ `imageFile` est une référence ; on copie la référence, l'idbMiddleware persistera correctement (chaque projet a son propre record `files`). Tester explicitement.
- **Defaults de pose** : centraliser `DEFAULT_POSE_PARAMS` dans `core/types.ts` ou dans `core/defaults.ts` pour éviter duplications. Probablement déjà existant — à confirmer en lecture.
- **Cascade checkboxes** : risque de logique embrouillée. Modèle simple : `rows ⇒ rooms ⇒ ∅` et `rows ⇒ catalog`. Implémenter via state local + helper `applyDependencies(options)`.
- **Performance** : pour des projets de taille raisonnable (< 100 pièces), `cloneProject` est négligeable. Pas d'optimisation prématurée.
