# Sous-étape 13.1 — Pose auto : respecter `minPlankLength`

## 1. Objectif utilisateur

Lorsque l'utilisateur ajoute une rangée ou drag une rangée précédente qui déclenche une cascade, le xOffset par défaut n'est plus calculé uniquement pour consommer toute la chute précédente : il est **borné** pour que la dernière lame soit ≥ `minPlankLength` quand c'est possible. Concrètement, plus de violations évitables de dernière lame courte après une pose automatique.

Le drag manuel d'un segment n'est **pas** contraint — l'utilisateur reste libre de positionner comme il veut. Seule la logique **automatique** (`computeDefaultXOffset`) est raffinée.

## 4. Comportement attendu

- [ ] Si la pose par défaut (reuse total de la chute précédente) produit une dernière lame ≥ `minPlankLength` → comportement inchangé.
- [ ] Sinon, `computeDefaultXOffset` renvoie la plus grande valeur `x ∈ [0, xOffset_full]` telle que la dernière lame satisfait `lastPlank.length ≥ minPlankLength`.
- [ ] Si aucune valeur dans `[0, xOffset_full]` ne satisfait (cas pathologique : largeur pièce très grande ou très petite), retomber sur `xOffset = 0` (planche entière au début, pas de reuse).
- [ ] La cascade (`propagateOffcuts`) applique la même logique puisqu'elle délègue à `computeDefaultXOffset`.
- [ ] Le drag manuel d'un segment **n'applique pas** cette logique (l'utilisateur pose ce qu'il veut, les violations restent visibles).
- [ ] Les `OffcutLink` reflètent la réalité : si la chute n'est plus consommée intégralement, aucun lien n'est produit vers la source (la correspondance `Math.abs(offcut - consumed) < EPSILON` reste la règle). La chute « résiduelle » (non consommée) est considérée perdue côté domaine — ne pas raffiner `computeOffcutLinks` dans cette sous-étape.

## 5. Calcul — notation et formules

Soient :
- `L = plankType.length`, `W = roomWidth - 2·cale`, `m = minPlankLength`.
- `xOffset_full = L - prevOffcut` (candidat initial, comportement actuel).

Pour une valeur `x ∈ [0, L)`, `fillRow` produit :
- Première lame : `first = min(L - x, W)` si `x > 0`, sinon `L`.
- Lames pleines suivantes, tant que `cursor + L ≤ W - EPSILON`.
- Dernière lame : le reste éventuel, de longueur `lastLen ∈ (0, L)`.

**Cas triviaux** :
- Si `W < L - x` → la rangée tient en une seule lame tronquée ; pas de « dernière lame courte » au sens des violations (elle est *la* lame, gérée par le cas `first-plank-too-short` dont on ne parle pas ici).
- Si `lastLen == 0` (division exacte) → pas de violation à corriger, garder `x = xOffset_full`.

**Cas général** (`W ≥ L - x` et `lastLen > 0`) :
- La dernière lame mesure `((W - (L - x)) mod L)` si `x > 0`, sinon `(W mod L)`.
- Contrainte : `lastLen ≥ m`.
- Équation : `(W - L + x) mod L ≥ m`.
- Sachant que faire varier `x` de `x_full` vers 0 décale linéairement le modulo, on cherche le plus grand `x ≤ x_full` tel que la condition tient.

**Implémentation recommandée** : recherche par simulation (appel `fillRow`) sur une grille fine (pas 0,1 cm, aligné sur la précision domaine). Simple, lisible, testable, sans optimisation prématurée. La grille balaie `x_full → 0` par pas de `0.1` et retourne le premier `x` satisfaisant. Complexité bornée par `L / 0.1` ≈ 1200 itérations max, fillRow O(W/L) → acceptable.

**Alternative** : calcul direct par arithmétique modulaire (rapide mais plus délicat à relire). À garder en fallback si la simulation pose un problème de perf mesurable (non anticipé).

## 6. Tests

### Unitaires — `add-row.feature`

Scénarios additionnels :

```gherkin
Scénario: Ajout d'une rangée — reuse complet ne viole rien → comportement inchangé
  Étant donné une pièce rectangulaire 400 cm × 280 cm
    Et un type de lame "Chêne" 120×14 cm
    Et une rangée existante avec xOffset 0 (chute finale = 40 − sawWidth)
    Et minPlankLength = 30
  Quand j'ajoute une rangée du même type
  Alors le xOffset de segment 0 consomme toute la chute (xOffset = 120 − 39,9 = 80,1)
    Et la dernière lame mesure >= minPlankLength

Scénario: Ajout d'une rangée — reuse complet violerait minPlankLength → xOffset borné
  Étant donné une pièce rectangulaire 250,5 cm × 280 cm
    Et un type de lame "Chêne" 120×14 cm
    Et une rangée existante avec xOffset 0 (chute finale = 10 − sawWidth)
    Et minPlankLength = 30
  Quand j'ajoute une rangée du même type
  Alors le xOffset est inférieur à xOffset_full (reuse borné)
    Et la dernière lame mesure >= minPlankLength
    Et aucun OffcutLink n'est créé vers la rangée précédente

Scénario: Ajout d'une rangée — aucune valeur ne satisfait → fallback xOffset 0
  Étant donné une configuration où aucun xOffset ∈ [0, xOffset_full] n'évite la violation
  Quand j'ajoute une rangée
  Alors xOffset = 0
    Et la rangée démarre par une lame entière
```

### Unitaires — `propagate-offcuts.feature`

Scénario supplémentaire :

```gherkin
Scénario: Cascade post-drag borne la reuse pour respecter minPlankLength
  Étant donné trois rangées du même type avec cascade automatique
  Quand je drag le segment 0 de la rangée 1 à un xOffset qui, propagé, produirait une dernière lame < minPlankLength sur la rangée 2
  Alors après release, le xOffset de la rangée 2 est borné
    Et la dernière lame de la rangée 2 est >= minPlankLength
```
