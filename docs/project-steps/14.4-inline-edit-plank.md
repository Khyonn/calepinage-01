# Sous-étape 14.4 — Édition inline de la longueur de planche depuis le canvas

## 1. Objectif utilisateur

Sur tout segment d'une rangée, la **première** et la **dernière** planches portent en permanence une annotation chiffrée (longueur en cm), même quand elles sont de longueur nominale (planches pleines). L'utilisateur **double-clique** sur l'annotation pour la transformer en champ éditable, tape une nouvelle longueur, et la perte de focus (blur) commit la valeur — Entrée commit aussi, Échap annule. Le `xOffset` du segment est recalculé ; la cascade `propagateOffcuts` se déclenche comme pour un drag. Cas d'usage : ajustement précis au mm près, là où le drag souris est limité par la résolution écran.

## 5. Calcul — formules domaine

Soient `L = plankType.length`, `W = segWidth`, `xOffset` courant ≥ 0.

**Édition de la première — formule directe** :
- `xOffsetFromFirstLength(f_new, plankType): number`
- `xOffset_new = L - f_new`.
- Clamp : `0 ≤ xOffset_new < L`. Si `f_new ≥ L` → `xOffset_new = 0` (planche pleine).
- Si `f_new ≤ 0` → clamp à `xOffset = L - 0.1` (sliver minimal, violation visible, utilisateur libre).

**Édition de la dernière — recherche par simulation** :
- `xOffsetFromLastLength(ℓ_new, plankType, segWidth, poseParams): number`
- Itère `xOffset` ∈ `[0, L)` par pas de 0,1 cm, calcule `last = fillRow(xOffset, segWidth, plankType, poseParams).last.length` à chaque pas, retient celui qui minimise `|last - ℓ_new|`.
- En cas d'égalité (plusieurs xOffset donnent même `|delta|`), retenir le **plus petit** xOffset (perte minimale en début de rangée).
- Range exhaustif (≈ 1200 itérations max), `fillRow` en O(W/L) → coût acceptable, pas d'optimisation prématurée. Cohérent avec le pattern bornage 13.4.
- Avantage vs formule directe : robuste quand la dernière était pleine (`ℓ_orig = L`), quand l'édition traverse un boundary (`n` change), ou quand la formule donnerait un résultat hors range.

**Pourquoi pas formule directe pour la dernière** : la formule `xOffset_new = xOffset_orig + (ℓ_new − ℓ_orig)` ne tient qu'à `n` constant. Quand la dernière est pleine au départ, ou quand l'édition traverse un boundary, le résultat est imprévisible. La simulation gère tous les cas uniformément.

## 8. Risques & arbitrages

- **`<foreignObject>` cross-browser** : Firefox a historiquement des bugs sur le rendu d'inputs HTML dans foreignObject zoomé. Test manuel sur Firefox + Chrome avant clôture. Fallback possible : overlay HTML hors SVG positionné via `getBoundingClientRect()` (même pattern que Tooltip 14.1).
- **Conflit double-click vs drag** : le `pointerdown` sur annotation doit `stopPropagation` pour ne pas démarrer un drag du segment. Le double-click natif (`onDoubleClick`) gère le timing 2-clics ; pendant le délai entre les deux clics, le drag risque de démarrer si on n'arrête pas la propagation. Vérifier `useSegmentDrag` ne déclenche que sur les `<rect>` planche, pas sur les `<g>` / `<foreignObject>` annotation.
- **Précision 0,1 cm** : commit arrondi (cohérent avec drag). Documenter dans le placeholder ou la doc.
- **Annotations toujours visibles** : changement de comportement par rapport au filtre actuel `length < plankType.length`. Vérifier que ça ne pollue pas visuellement les pièces longues (beaucoup de rangées, beaucoup de chiffres). Si problème → introduire option de masquage, hors scope 14.4.
- **Simulation pour dernière** : ~1200 itérations × `fillRow` O(W/L), exécution synchrone côté client. Pas de souci perf. Cohérent avec bornage.
- **Mobile / tactile** : double-tap pour l'édition, faisable. Pas testé en priorité ce jalon.
