# Spec 16.6 — Undo/redo via `redux-undo`

## Contexte & objectif

Session S6 du jalon 16. Brancher un historique undo/redo sur le store, avec :

1. Wrap **uniquement** du `projectReducer` via `redux-undo` (pas le `uiSlice`).
2. Filtrage strict : seules les actions métier `project/*` produisent une étape d'historique.
3. Consolidation `groupBy` pour `setRoomYOffset` (live dispatch — sinon spam d'entrées undo).
4. Raccourcis clavier `Ctrl+Z` / `Ctrl+Shift+Z` / `Ctrl+Y` au niveau document.
5. **Ctrl+Z contextuel** en mode `draw` : si un tracé est en cours, supprime le dernier point local au lieu d'undo Redux.
6. UI dédiée : composant `<UndoRedoButtons />` dans la `Topbar`, états `disabled` selon `past.length` / `future.length`.

Les sessions S1-S5 ont posé les pré-requis : reducers purs, drags `preview pendant / commit au release` (un drag = une action), schéma IDB allégé. L'historique reste donc sain par construction.

## Détails par changement

### A. Wrap `projectReducer` via `redux-undo`

#### A.1 Fichier `src/store/undoableProject.ts`

```ts
import undoable, { includeAction, groupByActionTypes } from 'redux-undo'
import type { AnyAction } from '@reduxjs/toolkit'
import { projectReducer, projectActions } from '@/store/projectSlice'

const UNDOABLE_ACTION_TYPES: string[] = [
  projectActions.updateName.type,
  projectActions.updatePoseParams.type,
  projectActions.addRoom.type,
  projectActions.updateRoom.type,
  projectActions.setRoomYOffset.type,
  projectActions.deleteRoom.type,
  projectActions.addRow.type,
  projectActions.updateSegmentOffset.type,
  projectActions.deleteRow.type,
  projectActions.addPlankType.type,
  projectActions.updatePlankType.type,
  projectActions.deletePlankType.type,
  projectActions.setPlan.type,
  projectActions.updatePlan.type,
  projectActions.removePlan.type,
]

const SET_ROOM_YOFFSET = projectActions.setRoomYOffset.type

function groupByYOffset(action: AnyAction) {
  if (action.type !== SET_ROOM_YOFFSET) return null
  const id = (action as ReturnType<typeof projectActions.setRoomYOffset>).payload.id
  return `${SET_ROOM_YOFFSET}/${id}`
}

export const undoableProject = undoable(projectReducer, {
  limit: 50,
  filter: includeAction(UNDOABLE_ACTION_TYPES),
  groupBy: groupByYOffset,
  syncFilter: true,
  ignoreInitialState: true,
})
```

**Pourquoi `includeAction` (et pas `excludeAction`)** :
- Liste exhaustive des actions métier connues, court (~15 entrées).
- Toute action UI est ignorée par construction (pas dans la liste).
- Toute action interne `redux-undo` (`@@redux-undo/UNDO`, `/REDO`, `/CLEAR_HISTORY`, `/INIT`) est ignorée par construction.
- Les actions `project/open`, `project/close`, `project/create`, `project/delete` **ne sont pas dans la liste** : changer de projet, créer ou supprimer un projet **ne** doit **pas** être undoable. Voir A.4.

**Pourquoi `groupBy` uniquement sur `setRoomYOffset`** :
- `setRoomYOffset` est dispatché en live pendant l'appui sur les flèches du `<YOffsetField>` (cf. `.agent/conventions.md` §Forms cas 2).
- Sans groupBy, un appui prolongé crée 30 entrées d'historique.
- TTL : `groupByActionTypes` (helper officiel) ne supporte pas un TTL natif ; on retourne une clé combinée `type+roomId`. La clé reste stable tant que les dispatches s'enchaînent. Quand la rangée Y change pour autre raison (autre action interleave), la clé est différente → nouveau groupe. Comportement souhaité.

> **Note TTL ~500ms** : `redux-undo` ne fournit pas nativement un debounce/TTL. Stratégie retenue : la clé stable suffit en pratique car les dispatches `setRoomYOffset` consécutifs sur la même room sont contigus (pas d'autre action métier interleavée pendant un appui clavier). Si l'expérience révèle une consolidation trop agressive (ex. yOffset modifié à T0, puis à T+10s, fusionnés en une seule entrée), on bascule vers une approche custom : middleware qui fait flush `@@redux-undo/INSERT` après 500ms d'inactivité. Reporter cette amélioration à S7+ si nécessaire.

import { listProjects, loadProject } from '@/persistence/db'

  localStorageLastProjectId,
  type LastProjectIdStore,
} from '@/persistence/lastProjectId'

const rootReducer = combineReducers({
  project: undoableProject,
  ui: uiReducer,
})

interface CreateAppStoreOptions {
  lastProjectIdStore?: LastProjectIdStore
}

export async function createAppStore({
  lastProjectIdStore = localStorageLastProjectId,
}: CreateAppStoreOptions = {}) {
  const list = await listProjects()
  const lastId = lastProjectIdStore.read()

  let current = null
  if (lastId && list.some(p => p.id === lastId)) current = await loadProject(lastId)
  else if (list.length > 0) current = await loadProject(list[0].id)

  const store = configureStore({
    reducer: rootReducer,
    preloadedState: {
      project: { past: [], present: { list, current }, future: [], _latestUnfiltered: null, group: null, index: 0, limit: 50 },
    },
    middleware: (getDefaultMiddleware) =>
      getDefaultMiddleware({

          ignoredPaths: ['project.present.current.backgroundPlan.imageFile'],
          ignoredActionPaths: ['payload.imageFile'],
        },
        immutableCheck: {
          ignoredPaths: ['project.present.current.backgroundPlan.imageFile'],

  const resolved = store.getState().project.present.current

### C. `idbMiddleware` — sauvegarder aussi sur undo/redo

Le `present` change après `@@redux-undo/UNDO` ou `/REDO`. La persistance IDB doit refléter le state effectif.

Réécriture :

```ts
import type { Middleware } from '@reduxjs/toolkit'
import { ActionTypes as UndoActionTypes } from 'redux-undo'
import type { AppState } from '@/store/types'
import { isProjectAction } from '@/store/types'
import { projectSlice } from '@/store/projectSlice'
import { saveProject, deleteProject } from '@/persistence/db'

const UNDO_TYPES = new Set<string>([
  UndoActionTypes.UNDO,
  UndoActionTypes.REDO,
])

export const idbMiddleware: Middleware<object, AppState> = (store) => (next) => (action) => {
  const result = next(action)

  const isUndoRedo = typeof action === 'object' && action !== null
    && 'type' in action && typeof (action as { type: unknown }).type === 'string'
    && UNDO_TYPES.has((action as { type: string }).type)

  if (!isProjectAction(action as { type: string }) && !isUndoRedo) return result

  const { project } = store.getState()
  const present = project.present

  if (projectSlice.actions.delete.match(action as never)) {
    void deleteProject((action as ReturnType<typeof projectSlice.actions.delete>).payload.id)
  } else if (present.current) {
    void saveProject(present.current)
  }

  return result
}
```

### D. Hook `useUndoShortcuts`

`src/hooks/useUndoShortcuts.ts` :

```ts
import { useEffect } from 'react'
import { ActionCreators } from 'redux-undo'
import { useAppDispatch } from '@/hooks/redux'

const FORM_SELECTOR = 'input, textarea, select, [contenteditable]'

function focusInsideForm(): boolean {
  const el = document.activeElement as HTMLElement | null
  return !!el && typeof el.matches === 'function' && el.matches(FORM_SELECTOR)
}

interface Args {
  /**
   * Si fourni, appelé en priorité quand `Ctrl+Z` est pressé.
   * Si retourne `true`, l'undo Redux n'est pas dispatché.
   * Permet à un mode (ex. draw) de gérer un undo local (supprimer dernier point).
   */
  onContextualUndo?: () => boolean
}

export function useUndoShortcuts({ onContextualUndo }: Args = {}) {
  const dispatch = useAppDispatch()

  useEffect(() => {
    const handler = (e: KeyboardEvent) => {
      const ctrlOrMeta = e.ctrlKey || e.metaKey
      if (!ctrlOrMeta) return
      if (focusInsideForm()) return

      const isUndo = e.key.toLowerCase() === 'z' && !e.shiftKey
      const isRedo =
        (e.key.toLowerCase() === 'z' && e.shiftKey) ||
        e.key.toLowerCase() === 'y'

      if (isUndo) {
        if (onContextualUndo?.() === true) {
          e.preventDefault()
          return
        }
        e.preventDefault()
        dispatch(ActionCreators.undo())
        return
      }
      if (isRedo) {
        e.preventDefault()
        dispatch(ActionCreators.redo())
        return
      }
    }
    window.addEventListener('keydown', handler)
    return () => window.removeEventListener('keydown', handler)
  }, [dispatch, onContextualUndo])
}
```

**Branchement** :

- `App.tsx` : `useUndoShortcuts()` (sans callback). C'est le fallback global.
- `DrawOverlay/index.tsx` : `useUndoShortcuts({ onContextualUndo })` avec un callback qui supprime le dernier vertex local et retourne `true`. Si pas de tracé en cours, retourne `false` → le hook global de `App.tsx` s'occupe de l'undo Redux.

> **Attention double listener** : monter `useUndoShortcuts` à deux endroits ajoute deux listeners. Si le premier listener (DrawOverlay) `preventDefault` mais ne `stopPropagation` pas, le second (App) reçoit aussi l'event et dispatch un undo Redux. **Solution** : appeler `e.stopImmediatePropagation()` dans `DrawOverlay` quand `onContextualUndo` retourne `true`. Le hook ne fait pas ça par défaut → expose plutôt un handler explicite dans `DrawOverlay` (voir E).

### E. `DrawOverlay` — Ctrl+Z contextuel

`useDrawInteraction.ts` actuel écoute `Backspace`/`Delete` pour supprimer le dernier point. **Conserver** ce comportement existant.

**Ajouter** : exposer une fonction `popLastVertex(): boolean` qui retourne `true` si un vertex a été supprimé, `false` si la liste était vide.

```ts
// useDrawInteraction.ts — ajouter dans le return
const popLastVertex = (): boolean => {
  if (verticesRef.current.length === 0) return false
  setVertices(verticesRef.current.slice(0, -1))
  return true
}

return { /* …, */ popLastVertex }
```

`DrawOverlay/index.tsx` :

```tsx
const { /* …, */ popLastVertex } = useDrawInteraction({ /* … */ })
useUndoShortcuts({
  onContextualUndo: () => popLastVertex(),
})
```

**Cas limite** : si l'utilisateur est en mode `draw` mais qu'aucun vertex n'a été posé, `popLastVertex()` retourne `false` → le hook tombe en undo Redux global. Comportement souhaité.

**Pas de double listener** : `App.tsx` monte `useUndoShortcuts()` global. `DrawOverlay` (monté à l'intérieur de l'arbre `App`) monte une seconde instance avec `onContextualUndo`. Les deux listeners sont actifs simultanément. Pour éviter le double dispatch :

- Le listener du `DrawOverlay` est ajouté **après** celui de `App` (ordre de mount).
- Quand `Ctrl+Z` est pressé, **les deux** handlers sont appelés (dans l'ordre d'enregistrement).
- Si `App` dispatch d'abord undo Redux, puis `DrawOverlay` supprime un vertex → comportement faux.

**Solution retenue** : monter `useUndoShortcuts` **uniquement** dans `App.tsx`, et faire passer le mode + callback par contexte. Concrètement, le hook lit `mode` et `popLastVertex` depuis un contexte `DrawContext` :

```ts
// DrawContext.tsx
import { createContext, useContext } from 'react'

interface DrawContextValue {
  popLastVertex: () => boolean
}

export const DrawContext = createContext<DrawContextValue | null>(null)
export const useDrawContext = () => useContext(DrawContext)
```

`DrawOverlay` provider la valeur. `App.tsx` :

```tsx
useUndoShortcuts({
  onContextualUndo: () => drawCtxRef.current?.popLastVertex() ?? false,
})
```

Trop indirect. **Solution simplifiée** : un seul `useUndoShortcuts` au niveau `App`. Le hook lit le `mode` Redux directement et, en mode `draw`, appelle un callback enregistré via une ref module-level :

```ts
// useUndoShortcuts.ts
const contextualHandlers = new Set<() => boolean>()

export function registerContextualUndo(fn: () => boolean) {
  contextualHandlers.add(fn)
  return () => contextualHandlers.delete(fn)
}
```

```ts
// DrawOverlay/index.tsx
useEffect(() => {
  return registerContextualUndo(popLastVertex)
}, [popLastVertex])
```

Le hook global itère sur `contextualHandlers` jusqu'à ce qu'un retourne `true`. Si aucun, dispatch undo Redux.

> **Décision finale** : option « set module-level + register ». Simple, testable, pas de prop drilling, pas de double listener. Documenter dans le hook qu'il est singleton.

## Rollback / risques

- **Risque API `redux-undo`** : version installée potentiellement différente sur les exports `ActionTypes` / `ActionCreators` / `StateWithHistory`. Vérifier `node_modules/redux-undo/lib/index.d.ts` après install. Adapter les imports.
- **Risque `preloadedState` mal formé** : si les 7 champs internes manquent au boot, `redux-undo` peut crash silencieusement (état invalide). Utiliser le helper `emptyHistory()` partout (store init, tests).
- **Risque double listener clavier** : géré via `registerContextualUndo` module-level (cf. E). Tester explicitement le scénario draw + Ctrl+Z.
- **Risque sélecteurs oubliés** : si un sélecteur lit encore `state.project.X` sans `.present`, le composant recevra `undefined` ou `past`. Faire le grep de contrôle systématiquement.
- **Risque fuite IDB** : si `idbMiddleware` n'écoute pas les actions `@@redux-undo/UNDO` / `/REDO`, les undo/redo ne sont pas persistés. Au prochain reload, l'IDB contient l'état pré-undo. **Bloquant** : test manuel point 12 obligatoire.
- **Risque limite 50 trop bas** : pour un projet complexe avec beaucoup de drags, 50 est rapidement atteint. Acceptable pour une première version. Si feedback utilisateur, monter à 100.
- **Risque actions `project/open` filtrées** : par construction, `project/open` n'est pas dans `UNDOABLE_ACTION_TYPES` → ne crée pas d'étape. Mais `redux-undo` continue d'appliquer le reducer (le state change). Vérifier que le reducer est bien appelé même quand l'action est filtrée (oui — `filter` contrôle uniquement la création d'une entrée d'historique, pas l'application au reducer).
- **Risque ordre middleware** : `idbMiddleware` doit être après le wrapping `redux-undo`. Avec `combineReducers` et middleware standard, l'ordre est correct par défaut.
- **Risque test flaky `groupBy`** : si la clé de groupage repose sur le `roomId` et que les tests utilisent toujours le même UUID fixture, le groupe persiste. Tester aussi avec deux rooms différentes pour vérifier que les dispatches sur `room A` puis `room B` créent **deux** entrées (pas groupés).
